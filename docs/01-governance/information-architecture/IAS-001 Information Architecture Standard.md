# IAS-001 — Information Architecture Standard

## 1. Purpose
This standard defines the permanent Repository Information Architecture for the HC-AI SDLC v1.0 framework. It establishes the conceptual domains, systems, and artifacts, defining their boundaries, ownership, and inheritance of authority. This document ensures that the repository structure remains understandable, maintainable, and aligned with governance as the framework scales.

## 2. Core Principles
- **Governance by Domain**: The repository structure strictly mirrors the governance hierarchy through explicitly defined domains, rather than generic function-based folders.
- **Primary Responsibility Placement**: A document belongs to the domain that owns its primary responsibility, not the domain that references or consumes it.
- **Separation of Evidence**: Authoritative framework knowledge (`docs/`) is strictly separated from evidence of execution (`records/`) and consuming entities (`products/`).

## 3. Taxonomy of Information

### 3.1. Governance Domain (Level 1)
- **Definition**: The highest tier of authority. It establishes the universal laws, constitutional charters, meta-standards, and structural rules that govern all downstream activities.
- **Authority**: Dictates the absolute constraints and definitions for all lower levels.
- **Examples**: Universal Constitution, Information Architecture Standard (IAS), Governance Hierarchy Specification (GHS), Terminology Standard (GLOSSARY).
- **Location**: `docs/01-governance/`

### 3.2. Operational Domain (Level 2)
- **Definition**: Translates governance principles into abstract processes, lifecycles, and operational workflows. It defines *how* work should flow without dictating the concrete tools used to execute it.
- **Authority**: Constrained by Level 1; governs Level 3.
- **Examples**: Operational Model, overarching lifecycle phases.
- **Location**: `docs/02-operational/`

### 3.3. Systems Domain (Level 3)
- **Definition**: The conceptual systems that implement the operational processes. Systems define standards, rules, constraints, and procedures for specific capability domains (e.g., Engineering, Verification, Specification).
- **Authority**: Constrained by Level 2; governs Level 4.
- **Examples**: Engineering Standards (ES), Repository Standards (RP), Research Standards (RS).
- **Location**: `docs/03-systems/`

### 3.4. Implementation Domain (Level 4 - EOS)
- **Definition**: The tangible tools, templates, platforms, and scaffolding that physically execute the framework. EOS (Engineering Operating System) is the primary implementation of the Operational Model.
- **Authority**: Constrained by Level 3; provides the physical assets for Level 5.
- **Examples**: Reusable assets, standard templates, starter kits, agent session managers.
- **Location**: `docs/04-eos/`

### 3.5. Consumers (Products - Level 5)
- **Definition**: The end products and project implementations that consume, adopt, and abide by the HC-AI SDLC framework.
- **Authority**: Bound by the entire framework. Product-specific decisions do not influence framework governance.
- **Examples**: QuizArena, AI PROOF, product-specific codebases and documentation.
- **Location**: `products/`

### 3.6. Evidence (Records)
- **Definition**: Ephemeral, point-in-time, or historical proof of execution. Records are not governance; they are the output generated by following the framework.
- **Authority**: Zero governing authority. They are descriptive, not prescriptive.
- **Examples**: Architecture Decision Records (ADRs), Validation Reports, Lessons Learned, Observations.
- **Location**: `records/`

## 4. Inheritance and Constraints

1. **Downward Inheritance**: Authority flows strictly top-down. Systems must comply with the Operational Domain; the Operational Domain must comply with the Governance Domain.
2. **No Upward Leakage**: Lower domains (e.g., Systems, EOS) cannot define rules that override or conflict with higher domains. Implementation details must never leak into Governance documents.
3. **Horizontal Independence**: Systems (e.g., Engineering System vs. Verification System) are horizontally independent but share a common parent domain. They must not create competing rules.

## 5. Document Placement Protocol

When creating or migrating a document, use the following decision matrix to determine its location:

- Is it a universal law, charter, or meta-standard? → **Governance Domain**
- Does it describe the abstract end-to-end lifecycle or operating model? → **Operational Domain**
- Is it a rule, standard, or procedure for a specific capability (e.g., coding, testing)? → **Systems Domain**
- Is it a template, script, tool, or starter kit? → **Implementation Domain (EOS)**
- Is it an output of an execution (a log, a decision, an observation)? → **Records**
- Is it an independent project built using the framework? → **Products**
